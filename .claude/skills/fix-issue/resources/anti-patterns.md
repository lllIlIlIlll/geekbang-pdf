# 修复反模式

---

## 速查表

| ❌ 禁止 | ✅ 应该 | 后果 |
|--------|--------|------|
| `catch(e) {}` 吞异常 | 消除导致异常的条件 | 问题被掩盖，下次更难排查 |
| 调用处加 null 判断掩盖问题 | 修复本不应返回 None/null 的函数 | 症状扩散到整个代码库 |
| 可变配置硬编码进业务代码 | 外置到配置文件，通过接口读取 | 依赖变更时需改多处 |
| 抛出不符合项目规范的裸异常 | 使用项目已有异常类层次和 error code | 错误信息格式不一致 |
| 说"修好了"不运行检查 | 运行检查，粘贴实际输出 | 建立错误信心，延误修复 |
| 修 Bug 顺手重构 | 先修 Bug，重构另开 PR | 回滚困难；掩盖真正修复点 |
| 修完不更新 CLAUDE.md | 每次必须沉淀经验 | 团队反复踩同一个坑 |
| 无限次自动重试 | 最多 3 轮，停下汇报卡点 | 消耗时间在错误方向上 |

## 症状 vs 根因

**判断方法：删掉修复后 Bug 若复现 → 你只修了症状。**

| 症状修复（错） | 根因修复（对） |
|-------------|-------------|
| 每个调用处加 `if result is None: return {}` | 修复函数本身为何返回 None |
| `except SomeError: pass` 跳过异常 | 修复触发该异常的根本条件 |
| 硬编码 `sleep(5)` 掩盖竞态 | 修复等待条件的判断逻辑 |
| 在结果层过滤掉异常数据 | 修复数据生成逻辑的根因 |

## 回归定位：git bisect

```bash
git bisect start
git bisect bad                        # 当前状态有问题
git bisect good <最后正常的 SHA>       # e.g. 上一个 release tag

# 每次检出后测试：
# 仍有问题 → git bisect bad
# 正常     → git bisect good
# 重复 6-8 次，自动定位到问题提交

git bisect reset
git show <问题提交 SHA>               # 根因在这个 diff 里
```

## 竞态问题放大

偶现 Bug（加了日志就消失）→ 并发触发强制复现：

```bash
# 通用并发压测模板
for i in $(seq 50); do
  {你的复现命令} &
done
wait
# 验证：期望 50 次全部成功，无错误输出
```
